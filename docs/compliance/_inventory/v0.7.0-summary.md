## 2. Five newly-documented defensive layers (Task I extension)

The originating brief enumerated 22 capabilities to verify. During deep-verification, Task I surfaced **5 additional substrate-level defensive layers** that were under-credited in the brief and would have appeared as `consumer_responsibility` or `partial_coverage` items in Task F's honest-limitations doc. Codegraph confirms they ship at v0.7.0 and structurally address NSA-enumerated concerns. They are folded into Task E's mapping as `structurally_addressed`:

### 2.1 `request_validator_input_validation`
**NSA concern addressed:** Tool parameter injection (real-world issue 9). **What it ships:** `pub struct RequestValidator` at `src/validate.rs:1027` is the canonical fluent validation surface every wire-entry layer (HTTP handlers, MCP tools, CLI subcommands) routes through. Pre-#966 the same `validate_id` + `validate_namespace` + `validate_agent_id` chains were duplicated across 87 HTTP routes + 74 MCP tools + 79 CLI subcommands; adding a new cross-field invariant required three audited per-surface edits. Post-#966 a single struct method covers all three surfaces with a typed `ValidationError { field, reason }` envelope. **This is the substrate's primary defense against the NSA-enumerated tool-parameter-injection class.**

### 2.2 `dos_multi_layer_defense`
**NSA concern addressed:** Denial of service (concern h). **What it ships (7 layers):**

| Layer | Mechanism | Anchor |
|---|---|---|
| K8 quota | `memory_quota_status` MCP + `/api/v1/quota/status` HTTP | `src/profile.rs:419` |
| Token budget | cl100k_base wire ceiling for `tools/list` | `tests/token_budget_guard.rs:65/75` |
| HMAC webhook | R3-S1.HMAC fail-CLOSED dispatch | `src/subscriptions.rs:791-803` |
| SSRF guard | DNS resolution + IP allowlist, fail-CLOSED | `src/subscriptions.rs:1070-1100`, #1053 |
| DLQ cap | `MAX_REPLAY_ATTEMPTS` bounds retry storms | `src/federation/push_dlq.rs:195` |
| HTTP body cap | `DefaultBodyLimit::max(2 * 1024 * 1024)` (2 MB) | `src/lib.rs:512` |
| Nonce replay | `AI_MEMORY_FED_REQUIRE_NONCE` strict mode | `src/federation/signing.rs:42`, #922 |

**Honesty note:** DoS hardening is perpetual — these seven layers raise the cost of attack but do not eliminate the threat class. Operators tune quotas + rate limits to their deployment.

### 2.3 `substrate_native_verify_family`
**NSA concern addressed:** CVE-2025-49596 (MCP-Inspector RCE, real-world issue). **What it ships:** three substrate-native inspection subcommands that do NOT use Anthropic's separate MCP-Inspector toolchain (the surface CVE-2025-49596 targeted):

- `ai-memory verify-reflection-chain` (`src/daemon_runtime.rs:373`) — walks the `reflects_on` edge graph backward and verifies every Ed25519 signature
- `ai-memory verify-signed-events-chain` (`src/daemon_runtime.rs:380` + `src/cli/verify_signed_events.rs`) — walks the V-4 cross-row hash chain (`prev_hash + sequence`) on the `signed_events` audit table
- `ai-memory verify-forensic-bundle` (`src/daemon_runtime.rs:390` + `src/forensic/bundle.rs`) — re-hashes every file in a procurement-grade evidence bundle, checks manifest signature, re-verifies every edge signature

Substrate operators using these are **not vulnerable to CVE-2025-49596** (which was in the separate MCP-Inspector toolchain, not the ai-memory substrate).

### 2.4 `mcp_client_attestation` (PARTIAL)
**NSA concern addressed:** Tool invocation path confusion (real-world issue 10). **What it ships:** the substrate captures `clientInfo.name` from the MCP `initialize` handshake (`src/mcp/mod.rs:1607-1611, 2410-2411`) and threads it through every downstream operation. Per-row provenance attribution to a specific client (Claude Code, Cursor, Cline, etc.) supports forensic reconstruction of which client made which call.

**Honesty note + v0.7.x gap-fix candidate:** This is PARTIAL server-identity attestation. The substrate currently captures the CLIENT's identity but does NOT publish a signed `serverInfo` response in the `initialize` handshake. A misconfigured client mounting multiple memory servers cannot cryptographically distinguish them. **Proposed v0.7.x follow-up:** add daemon-Ed25519-signed `serverInfo.signature` to the MCP initialize response so clients can pin-on-first-connect (TOFU). Closes tool-invocation-path-confusion at the substrate boundary.

### 2.5 `encryption_at_rest_sqlcipher`
**NSA concerns addressed:** Access control + token security + misconfigurations. **What it ships:** operators may build with `--features sqlcipher` to enable SQLite-level AES-256 encryption at rest. Under `AI_MEMORY_ENCRYPT_AT_REST=true`, every DB open path requires `AI_MEMORY_DB_PASSPHRASE` (or `--db-passphrase-file` with mode 0400 strict-permission check per #1055). Plain-sqlite opens are refused under this posture.

**Honesty note:** Switching the flag on against an existing plain DB does NOT encrypt it — operators must export → encrypted-init → import or use the sqlcipher CLI `ATTACH … KEY` recipe.

## 4. v0.7.x gap-fix candidates surfaced during Task I

Three substrate-level gaps that COULD be closed in v0.7.x with small new code (1–2 NHI sessions each). Documented here so future v0.7.x follow-up issues consume the right baseline:

1. **Daemon `serverInfo` Ed25519 signing at MCP initialize handshake** — closes tool-invocation-path-confusion at substrate boundary. Substrate already has `load_daemon_signing_key` (`src/governance/audit.rs:558`); just need to thread it into the initialize response. LOE: 1–2 NHI sessions.
2. **`Accept-Provenance: verbose` capability negotiation flag** — closes the output-poisoning consumer-responsibility gap by letting consumers opt into Form 4/5/6 provenance signals per-session without flipping the wire default (which would be a backwards-compat break). LOE: 1 NHI session.
3. **Per-namespace rate-limit dimension extension** — tightens K8 quota from `(agent_id)` to `(agent_id, namespace)` compound so a malicious agent operating across namespaces can't bypass per-namespace allotments. LOE: 1 NHI session.

These belong in `[Unreleased] — v0.7.x doc + substrate follow-ups` if the operator authorises the code work; otherwise they remain documented in [`docs/compliance/honest-limitations.md`](../honest-limitations.md) §"Identified gap-fix candidates" as the substrate's planned closures.
